## 4. Кэш и потеря данных — РЕШЕНО, важные правила

**История проблемы:** был service worker с cache-first → телефон/ноут показывали
старую версию. Я посоветовал чистить «данные сайтов» — это **стёрло базу IndexedDB**
(ошибка с моей стороны).

**Текущее решение (в `index.html`):** service worker **полностью убран**. При заходе
страница сама делает `getRegistrations().unregister()` + чистит **только Cache Storage**
(оболочку), **НЕ трогая IndexedDB/localStorage**. Один раз перезагружается на свежак.

**ПРАВИЛА (повторять пользователю):**
- ✅ Обновить версию: на ноуте **Cmd+Shift+R**; на iPhone — закрыть вкладку и открыть заново.
- ⛔️ **НИКОГДА не советовать чистить «cookie / данные сайтов»** — это удаляет базу
  (полигоны, пули, патроны лежат в IndexedDB).
- Обновления версии саму базу НЕ стирают (IndexedDB переживает обновления; DB
  upgrade v4→v5 только добавляет stores, ничего не удаляет).
- Долговременная страховка данных — **синхронизация с Яндекс.Диском** (Настройки →
  Я.Диск). Один токен на всех устройствах → общие данные + бэкап в облаке.

> Офлайн-режим (PWA) сейчас отключён вместе с SW. Если понадобится — вернуть SW
> с правильным версионированием (см. git-историю sw.js), но это вторично.

## 6. Главный экран /calc — текущая структура (стиль AB Quantum)

Сверху вниз:
1. **HUD-карточка решения** (sticky): Вертикаль / Горизонталь (mil + MOA + см) со
   стрелками → дистанция (− [N] +) → **строка деталей TOF · Энергия · Скорость · Mach**
   → кнопка «🎯 Промах».
2. **Закреплённый контроллер** (всегда виден, оранжевая рамка): переключатель
   **🧭 Компас / 🕐 Часы** → скорость ветра + виджет направления → кнопки захвата
   **🎯 Навёл на цель / 🌬 Откуда ветер** → V₀ + азимут → кнопки погоды (🌐/📡).
3. Вкладки **Профиль / Атмо / Прочее**. Вкладка «Профиль» (как в AB): селекторы
   оружие + патрон (выбор активного профиля), **сводка только для чтения**
   (BC/модель/масса/V₀/высота прицела/нолевая/твист) + кнопки «✎ Оружие/Патрон».
   Ручные поля BC/масса/высота прицела/нолевая показываются ТОЛЬКО если профиль не
   выбран. Параметры берутся из weapon/cartridge. `updateProfileView()` в /calc.
4. Кнопка **📋 Все дистанции** → отдельный экран `/calc-table` (таблица не висит на главном).
5. Раскрывашки «🔭 Прицел» и «🌡 Атмосфера и баллистика».

### Ветер — конвенция и модель (ВАЖНО при доработке)
- В интерфейсе — **«откуда дует»** (международный стандарт).
- Внутри скрытое поле `windDir` хранит **абсолютное «куда дует»** (TO) — это источник
  правды для солвера: `windAngle_deg = solverWindAngle(windDir, azimuth)`.
- Компас: оранжевая стрелка = «откуда» (= windDir+180 в отображении), серая = азимут цели.
- Часы (`createWindClock`): цель = 12:00 (встречный), 3=справа, 9=слева, 6=попутный.
- Захват компасом (`captureHeading()` + модуль `Compass` в `devices.js`):
  «🎯» пишет в `azimuth_deg`, «🌬» пишет «откуда» → конвертируется в windDir (+180).

### Компас телефона — калибровка
- iOS калибрует магнитометр **автоматически** при движении устройства; отдельной
  «кнопки калибровки» в вебе нет (нельзя вызвать программно).
- Если точность плохая — приложение читает `webkitCompassAccuracy` (модуль `Compass`,
  поле `state.accuracy`) и показывает подсказку: «поводи телефоном восьмёркой».
  Порог «плохо»: `accuracy == null || < 0 || > 20`.
- Мешают: MagSafe/магнитные чехлы, металл рядом. Совет пользователю — снять чехол,
  отойти от машины/железа, сделать «восьмёрку».

5. **Ротор-мишень (`/moving-target`, режим Ротор) — переделать метод расчёта.**
   Найдено при разборе реального кейса (ось горизонтальная направлена на стрелка,
   вал земли скрывает нижнюю половину дуги, гонг появляется слева/прячется справа):
   - Текущий расчёт даёт «упреждение в mil» и текст справки советует целиться
     «в стороне от 3 часов» — но в этой точке (3 часа = точка появления/скрытия
     у вала) движение ЧИСТО ВЕРТИКАЛЬНОЕ, а формулировка звучит как горизонтальный
     сдвиг. Путает пользователя, доверия к расчёту не было — обоснованно.
   - **Реальная механика цели (равные лопасти по кругу) допускает НАМНОГО
     проще и надёжнее: не считать mil-упреждение вообще, а считать ВРЕМЯ.**
     Видимая дуга = ровно полукруг (вал делит поровну). Интервал между
     соседними лопастями в ОДНОЙ точке = T/N (T — период оборота, N — кол-во
     лопастей). Время от появления слева до скрытия справа = T/2.
     Момент нажатия спуска = (T/N или T/2) − TOF − время_реакции, считая от
     видимого события (лопасть прошла точку / лопасть появилась).
     Это ровно то, что пользователь и делает на глаз в реальности («жду, пока
     один гонг скроется, и по ощущению жду следующий») — только с точным
     числом вместо интуиции.
   - **Единицы ввода**: пользователь хочет вводить радиус лопастей И диаметр
     гонга в ДВУХ единицах на выбор — метры (если известно с брифинга) ИЛИ
     mil (если мерить по сетке прицела визуально на дистанции). Нужен
     переключатель единицы на оба поля (radius_m/bladeSize_m), с конвертацией
     mil→м через дистанцию (м = mil/1000 × дистанция_м).
   - Количество лопастей (`blades`) — **уже редактируется** в текущем UI,
     ничего добавлять не нужно, только не путать пользователя, что этого нет.
   - Скорость вращения — пользователь сам замеряет секундомером (несколько
     оборотов, не один — точность), ввод RPM или периода T остаётся как есть.
   - **Уточнение после доп. исследования (мысленный отсчёт доли секунды
     ненадёжен — проверено через веб-поиск, разброс десятки мс даже у
     тренированных людей)**: вместо счёта времени (T/N, T/2) — выдавать
     **точку на сетке прицела** (в mil от фиксированной точки скрытия/появления
     у вала), на которую стрелок наводится и жмёт спуск ПРИ ВИЗУАЛЬНОМ
     пересечении гонгом этой метки — без счёта секунд вообще.
     Формула (аим-точка = точка скрытия, угол 0° от центра круга):
     ```
     ω = 2π/T
     φ = ω × (TOF + время_реакции)                 [рад, угол опережения]
     Δx = R × (cos(φ) − 1)   [м, смещение по горизонтали от аим-точки]
     Δy = R × sin(φ)         [м, смещение по вертикали от аим-точки]
     Δx_mil = Δx/дистанция × 1000,  Δy_mil = Δy/дистанция × 1000
     ```
     Направление знаков зависит от стороны вала и направления вращения —
     ОБА случая выведены и реализованы (для ccw знак Δx зеркально
     отражается: `dx_ccw = -dx_cw`, Δy не меняется — см. реализацию ниже).
     Проверено на реальном примере через сам солвер `ballistics.js` (не оценка):
     V₀=830 м/с, пуля 175gr, BC G7=0.261, R=1.5м, T=5с, N=4, D=400м,
     атмо стандартная (15°C/1013гПа/50%) → TOF=0.562с, реакция=0.2с (допущение,
     должно быть настраиваемым) → φ=54.8°, метка = **1.59 mil влево + 3.07 mil
     вверх** от точки скрытия за валом.
   - **Практическая реализация от пользователя (провалидирована, лучше просто
     вывода mil-точки в воздухе) — «трюк с накруткой + метка на ёлочке»**:
     большинство сеток не имеют меток НАД горизонтальной линией (только
     ниже, «ёлочка»), а нужна именно ДЛИННАЯ горизонтальная линия (чтобы
     поймать пересечение по любой горизонтали), которая физически есть
     только через центр перекрестия. Решение: докрутить барабанчик ВВЕРХ на
     Δy_mil СВЕРХ обычной баллистической поправки для этой дистанции, и
     удерживать точку у вала МЕТКОЙ «−Δy_mil» на ёлочке (не центром!). Смещение
     метки (−Δy) и доп.накрутка (+Δy) взаимно компенсируются для точки
     попадания пули (летит как при обычной поправке), но ВИЗУАЛЬНО переносят
     длинную горизонтальную линию перекрестия на нужную высоту. Стрелок смотрит
     на эту линию и жмёт спуск при пересечении её гонгом (пересечение будет
     ровно на Δx_mil от вертикали — работает как проверка, что поймали
     правильный момент). Критично: докрутка ДОБАВЛЯЕТСЯ к обычной поправке
     на дроп, а не заменяет её — иначе чистый промах по вертикали.
   - **✅ РЕАЛИЗОВАНО (коммит `fec6eca`)**. `/moving-target` → Ротор переписан:
     - `computeRow()` (было `computeTOF()`) — возвращает полную строку солвера
       (drop_mil/drift_mil/vel_mps), с применённым `applyCartridgeOffset` и
       `applySSF` — это и есть «реальная поправка по высоте с учётом дистанции».
     - Новые поля: `state.rotationDir` ('cw'|'ccw', UI-селектор «по часовой/против»)
       и `state.reactionSec` (число, поле ввода) — раньше не было направления
       вращения и времени реакции как настраиваемых параметров.
     - Формула Δx/Δy (см. выше) — реализована с учётом знака направления
       вращения: `dx_m = rotationDir==='ccw' ? -dxBase : dxBase` (для ccw знак
       зеркально отражается — выведено и проверено отдельно, см. рассуждение
       в истории сессии).
     - Экран показывает: базовую поправку (mil, из солвера) → доп. смещение
       (+Δy) → итоговую накрутку (сумма) → метку на ёлочке (−Δy) → точку
       пересечения (Δx, влево/вправо от вертикали) — расшифровка трюка
       «докрутка + метка» встроена текстом прямо в результат.
     - **SVG-схема сетки** (`drawRotorReticleDiagram(crossMil, holdMil)`) —

     краем», а не ось вращения).** Пользователь на реальной практике целится
     в точку, где гонг касается вала (берма) своим НИЖНИМ краем — визуально
     резкий, легко ловимый момент. Первая версия расчёта неявно считала
     Δx/Δy от идеализированной точки «центр гонга ровно на оси вращения»
     (θ=0) — а это НЕ то же самое: в момент касания нижним краем центр гонга
     находится выше оси на величину радиуса гонга (`r_gong` = половина
     диаметра), а не на оси. Разница на контрольном примере (R=1.5м,
     диаметр гонга 0.2м, T=5с, 400м) — угол `θ_ref = asin(r_gong/R_center)
     ≈ 4.1°`, что даёт ошибку ≈0.3 mil по вертикали и ≈0.1 mil по горизонтали
     — заметно для точной стрельбы, исправлено.

     от оси вращения до ВНЕШНЕГО края гонга (не до центра) — добавлена
     пометка под полем ввода в UI. Из этого радиуса и диаметра гонга
     выводится `R_center = radius_m − r_gong` (радиус до ЦЕНТРА гонга —
     именно то, что движется по кругу и что нужно поразить пулей).
     Новые формулы в `renderRotorResult()`:
     ```
     r_gong = bladeSize_m / 2
     R_center = radius_m − r_gong
     thetaRef = asin(r_gong / R_center)      // угол опорной точки от оси
     v_tan = ω × R_center                     // скорость ЦЕНТРА гонга (не внешнего края)
     dxBase = R_center × (cos(thetaRef+φ) − cos(thetaRef))
     dy_m   = R_center × sin(thetaRef+φ) − r_gong
     ```
     (было: `dxBase = R×(cosφ−1)`, `dy_m = R×sinφ` — частный случай при
     `thetaRef=0`, что справедливо только для гонга нулевого размера).
     Направление вращения (cw/ccw) отражается только на знаке `dx`, как и
     раньше — `thetaRef`/`dy` от направления не зависят (симметрия
     сохраняется). Проверено численно (Python) и сверено с реальным расчётом
     в приложении на контрольном примере — совпало точно (Δy=2.75 mil,
     Δx=1.69 mil влево на 400м).

## 8c. Кнопка «+» (FAB) пряталась под нижней навигацией — исправлено

  — пользователь: выглядело как «спрятанная платная фича, боимся показывать».

  маркеры оставались тем же размером на экране, только в правильных
  позициях. Решение: позиции маркеров (`px` — SVG-юнитов на 1 mil) не зависят
  от `zoom` (геометрия всегда верная), а РАЗМЕР маркеров (радиусы, толщина

  маркеров остаются константными на любом уровне увеличения. Проверено
  визуально на zoom=1 и zoom=3 — маркеры одного размера на экране, сетка
  при этом кратно детальнее.

## 8m. Баг калибровки сетки — вертикальный масштаб уходил в 0
Пользователь откалибровал реальное фото: центр + точка «1 mil вправо»
(чисто по горизонтали). В расчёте на HUD-экране метки 100м/200м легли
почти друг на друга у самого центра (визуально читалось как одна метка
«1000м» — на самом деле две подписи «100м»/«200м» накладывались).

**Причина.** `numInput('refMilV', ..., cal?.p1_mil_v ?? 5, ...)` — поле
вертикального смещения ссылочной точки по умолчанию было **5**, а не 0.
Пользователь тапнул точку с реальным нулевым вертикальным смещением, но
поле осталось на дефолтных «5». `reticleScale()` посчитал
`px_per_mil_v = dy/mv = (~0)/5 ≈ 0` — вертикальный масштаб обнулился,
и ЛЮБОЙ `drop_mil` (какой угодно) даёт `py ≈ cy` — все метки садятся на
одну горизонтальную линию у центра независимо от реальной дистанции.
Проверено численно (`reticleScale()` напрямую в консоли): старый дефолт →
`vy:0`; после фикса → `vy` через изотропный fallback совпадает с `hx`,
как и должно быть у нормальной сетки (шаг по вертикали и горизонтали
почти одинаковый).

**Исправлено:**
1. Дефолт `refMilV` `5` → `0`. Теперь калибровка одной точки только по
   горизонтали (как в этом случае) автоматически включает уже
   существовавшую в `reticleScale()` изотропную логику («если задана
   только одна ось — берём вторую как такую же») — раньше её глушил
   стухший дефолт.
2. **Проверка на явную ошибку калибровки** — после тапа ссылочной точки
   считаем реальный масштаб (`reticleScale`) и если `hx`/`vy` отличаются
   больше чем в 4 раза — добавляем к `calStateInfo` предупреждение
   «⚠️ Похоже на ошибку калибровки…». На реальной сетке шаг по осям почти
   одинаковый, сильный перекос почти всегда значит, что одно из двух
   mil-полей не соответствует реально тапнутой точке.
3. Справка (i) дополнена абзацем про эту конкретную ошибку.

**Пользователю нужно** зайти в свою сохранённую сетку («Сетки прицелов» →
открыть) и перетапнуть ссылочную точку (или просто поправить поле
вертикального смещения на 0, если точка реально без вертикального
сдвига) и сохранить заново — старая калибровка с `p1_mil_v:5` останется
в данных, пока не пересохранят.

## 8n. Скролл страницы не работал над фото сетки (renderReticleViewer)
Пользователь: на HUD-экране (`/calc` → «Сетка») скролл страницы «не
реагирует примерно до половины страницы, а после половины начинает
скроллить» — ровно там, где расположена картинка фото-сетки
(`.reticle-viewport`/`.reticle-canvas`, `renderReticleViewer()`).

**Причина.** Обработчик `touchmove` для pinch-zoom (`www/app.js`) вызывал
`e.preventDefault()` безусловно для ЛЮБОГО жеста двумя касаниями:
```js
if (e.touches.length === 2 && pinchStart) {
  e.preventDefault();   // всегда, даже если это не pinch
  ...
}
```
Двупальцевый скролл (трекпад Mac, некоторые Android-жесты) держит
расстояние между двумя касаниями почти постоянным — пальцы едут вместе в
одну сторону, а не расходятся/сходятся. Код не отличал это от настоящего
pinch — итог: любой двупальцевый скролл над этой картинкой блокировался
безусловным `preventDefault`, страница не скроллилась именно в зоне этого
элемента.

**Диагностика.** Автоматический тест через инструмент эмуляции скролла
в браузере зависал (таймаут 30 c) — но зависал ОДИНАКОВО и над картинкой,
и над обычным текстом страницы, что показало: зависание — ограничение
самого тест-инструмента в этой среде, не воспроизведение бага. Реальную
причину подтвердили точечно: смоделировали настоящие `TouchEvent` через
`new TouchEvent(...)`/`new Touch(...)` и проверили `event.defaultPrevented`
— до фикса жест «пальцы едут вместе» (эмуляция скролла) блокировался
(`true`), после фикса — не блокируется (`false`), а настоящий pinch
(пальцы расходятся) продолжает работать (`true`).

**Фикс** — сравниваем текущее расстояние между касаниями с начальным; если
изменение меньше 10px, это не pinch — выходим из обработчика БЕЗ
`preventDefault`, отдавая жест браузеру (обычный скролл/pan). Порог 10px
выбран так, чтобы не «просаживать» начало настоящего pinch (в первые
несколько px движения реального pinch расстояние тоже меняется мало, но
это не критично — pinch продолжает работать, только чуть позже
переходит в активную фазу).

## 9. Предпочтения пользователя (важно)
- **ЭТАЛОН — Applied Ballistics Quantum (мануал).** Делать «как в AB», не отступать
  от него без явной просьбы пользователя. Это основа всех решений по UX/логике.
- Язык — русский. Делать самому (не гонять в терминал). Лаконичность.
- Не технарь — объяснять по шагам «куда нажать».
- Референс UI — **Applied Ballistics Quantum** (мануал: см. HANDOFF.md §1,
  путь к PDF в Google Drive). Главный экран — крупное решение сверху, редактируемые
  параметры (ветер/атмо) в зоне большого пальца снизу.
- **Единый стандарт обозначения направлений — U/D/L/R.** Вверх/вниз — `U`/`D`
  (Up/Down), влево/вправо — `L`/`R` (Left/Right). Применяется везде, где
  показана поправка с направлением (не только на экране ротора): подписи
  полей ввода со знаком (например «+ R», «+ U»), результаты калибровки сетки,
  сдвиг базового патрона/CBA, любые будущие калькуляторы. Раньше эти же вещи
  писались словами («вверх», «влево») — непоследовательно от экрана к экрану.
  Причина запроса: пользователь хочет один узнаваемый стандарт на все
  расчёты, а не разные слова в разных местах. НЕ трогать: HUD `/calc`
  (там уже стрелки ▲▼►◄ через `dirArrow()` — это невербальный, уже
  однозначный способ, замена на буквы была бы шагом назад); угол ветра
  (компас в градусах — отдельная, самостоятельная система, не путать с
  U/D/L/R); lead на движущейся линейной цели («вперёд/назад» по направлению
  движения цели — это не экранная ось, а направление вдоль курса цели,
  может быть под любым углом, U/D/L/R не применимо).
