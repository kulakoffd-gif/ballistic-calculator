## 1. Что это за проект

**BalisticNote Pro** — баллистический калькулятор уровня Strelok Pro / Applied
Ballistics Quantum + журнал стрельб и релоадинга. Технически — один и тот же
код (Capacitor + vanilla JS, без сборщика) собирается в **два продукта**:
Android APK (нативное приложение) и **веб-версию** (PWA, работает в браузере
на любом устройстве, включая iPhone).

Эталон дизайна и функциональности — **Applied Ballistics Quantum** (официальный
мануал, 42 стр., прочитан полностью). Проект целенаправленно двигается «как в
AB», где это оправдано, и осознанно расходится там, где решение пользователя
было другим (см. `AB-COMPARISON.md` — живая таблица паритета/расхождений).

Пользователь — не программист, работает через объяснения «на пальцах», сам
пушить/собирать не умеет и не хочет — Claude делает git push/деплой/мониторинг
сам, отчитываясь короткими апдейтами.

```
Ballistic Calculator/

│                                  статусы, что можно/нельзя сделать и почему)
├── UNITS.md                    ← справочник единиц измерения (зафиксированы
│                                  константами в коде, не переключаемы)
├── SETUP.md                    ← инструкция локальной сборки APK
├── package.json                ← Capacitor-зависимости
├── capacitor.config.json       ← appId=com.balisticnote.pro, webDir=www
├── debug.keystore               ← фиксированный keystore для стабильной подписи APK
├── .github/workflows/
│   ├── build-android.yml       ← CI: собирает APK на каждый push
│   └── deploy-pages.yml        ← CI: публикует www/ на GitHub Pages на каждый push
├── assets/                     ← иконки приложения (source для @capacitor/assets)
└── www/                        ← ВЕСЬ код приложения (общий для APK и веб)
    ├── index.html               ← оболочка, топбар, нижняя навигация, боковое меню
    ├── app.js (~5000+ строк)    ← все роуты/экраны, вся UI-логика
    ├── ballistics.js            ← солвер: RK4, G1/G7/custom drag, Coriolis, spin drift и т.д.
    ├── storage.js               ← IndexedDB-обёртка + merge-синхронизация (mergeAll)
    ├── devices.js                ← Bluetooth (Kestrel, SIG KILO) + компас + Open-Meteo
    ├── backup.js                ← авто-бэкап + двусторонняя синхронизация с Я.Диском
    ├── yadisk.js                ← Yandex.Disk REST API клиент
    ├── units.js                  ← форматтеры единиц (константы, без UI-переключателя)
    ├── style.css                 ← вся стилизация
    └── manifest.webmanifest      ← PWA-манифест
```

Подробное описание каждого модуля, солвера и хранилища — в `HANDOFF.md`.

## 5. Открытые вопросы

1. **Формат памяти проекта: консолидировать или оставить раздельно?** Сейчас
   параллельно существуют `PROJECT_CONTEXT.md` (новый, по этому шаблону) и уже
   давно ведущиеся `WORKLOG.md`/`HANDOFF.md`. Решение пока не принято
   пользователем — использовать оба (этот файл как вход, остальные как
   детали) или влить всё в один файл. Текущий подход — первый вариант.
2. **GitHub PAT-токен вводится вручную каждую сессию** — неудобно и не
   зафиксировано нигде постоянно. Нужно решить: заводить ли долгоживущий
   секрет/deploy key, или пользователя устраивает текущий процесс.
3. **Netlify-сайт с таблицей сравнения не обновляется автоматически** —
   при изменении `AB-COMPARISON.md` нужно вручную пересобирать HTML и
   передеплоивать. Нужно решить: настроить ли Netlify на авто-деплой из
   GitHub-репозитория (тогда обновления будут приходить сами), удалить сайт
   после разового просмотра, или оставить как есть (ручной процесс).
4. **iOS-версия** — остаётся ли в планах? Заблокирована железом (см. §3),
   не переспрашивали давно.
5. **Наименования «Брифинг упражнения» vs «Target Card»** — недавно
   переименованы по прямому запросу пользователя, но исторически были похожи
   («PRS стейджи» vs «Цели (брифинг)») и раньше уже вызывали путаницу.
   Стоит перепроверить с пользователем, что новые названия однозначно понятны.
